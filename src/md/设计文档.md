# 需求背景
Appliance 是某云服务的云下插件，用于采集 vCenter 资源并将主机信息上报到云服务侧。用户在主机安装 Appliance 后，需要完成三阶段配置：
1. 输入云账号 AK/SK 与云服务域名（domain）。
2. 完成 Appliance 全局配置，并调用云侧注册接口获取并保存 `appliance_id`。
3. 完成 vCenter 配置并执行采集，用户从采集到的主机列表中选择部分主机（不超过 200 台）上报云侧。

# 核心需求
设计 Appliance 注册逻辑，明确云侧接口、数据库模型，以及“同一云账号下多 Appliance 的区分与管理”机制。

## 目标
1. 支持一个云账号（同一 AK/SK）绑定多个 Appliance。
2. 每个 Appliance 具备全局唯一且稳定的身份标识。
3. 注册接口幂等，避免重复创建/脏数据。
4. 支持基础运维能力：在线状态、停用。

## 关键约束
1. `AK/SK` 不能明文存储，必须加密或托管在密钥服务。
2. 重装/重启后，若是同一实例，应尽量返回同一个 `appliance_id`。
3. 重复注册同一实例不能产生多个 Appliance 记录。

# 需求设计（聚焦核心需求）
## 一、注册模型与主流程
1. Appliance 首次启动生成本地稳定标识 `instance_fingerprint`（建议由机器标识 + 安装随机盐计算哈希）。
2. 客户端调用注册接口提交 `AK/SK/domain/instance_fingerprint`。
3. 云侧先校验云账号合法性，再执行实例幂等判断：
   - 若存在相同 `(cloud_account_id, instance_fingerprint)`：返回已存在 `appliance_id`（重装恢复场景）。
   - 否则创建新 Appliance，分配 `appliance_id`。
4. 返回 `appliance_id`，客户端本地持久化。



## 二、同一云账号下多 Appliance 区分策略
1. 云账号维度：`cloud_account_id` 统一表示租户/云账号身份。
2. Appliance 维度：`appliance_id` 为全局唯一主键（UUID/雪花 ID 均可）。
3. 实例唯一性：`(cloud_account_id, instance_fingerprint)` 建唯一索引，区分同账号下多个实例。
4. 管理能力：每个 Appliance 维护状态机（`INIT/ACTIVE/OFFLINE/DISABLED`）和最后心跳时间。

## 三、数据库表设计
### 1) `tb_cloud_account`
|字段|类型|说明|
|---|---|---|
|id|bigserial PK|主键|
|account_ak|varchar(128)|云账号 AK（可脱敏展示）|
|account_sk_cipher|text|云账号 SK 密文|
|domain|varchar(256)|云服务域名|
|status|varchar(16)|`ACTIVE/LOCKED`|
|created_at|timestamp|创建时间|
|updated_at|timestamp|更新时间|

索引与约束：
1. 唯一索引：`uk_ak_domain(account_ak, domain)`。

### 2) `tb_appliance`
|字段|类型|说明|
|---|---|---|
|id|bigserial PK|主键|
|appliance_id|varchar(64)|对外 Appliance 标识（唯一）|
|cloud_account_id|bigint FK|关联 `tb_cloud_account.id`|
|instance_fingerprint|varchar(128)|实例指纹|
|hostname|varchar(128)|主机名（可选）|
|ip|varchar(64)|主机 IP（可选）|
|status|varchar(16)|`INIT/ACTIVE/OFFLINE/DISABLED`|
|last_heartbeat_at|timestamp|最后心跳时间|
|created_at|timestamp|创建时间|
|updated_at|timestamp|更新时间|

索引与约束：
1. 唯一索引：`uk_appliance_id(appliance_id)`。
2. 唯一索引：`uk_account_fingerprint(cloud_account_id, instance_fingerprint)`。
3. 普通索引：`idx_account_status(cloud_account_id, status)`。

## 四、接口设计
### 1) 注册接口
`POST /api/v1/appliances/register`

请求体：
```json
{
  "account_ak": "AK_xxx",
  "account_sk": "SK_xxx",
  "domain": "example.myhuaweicloud.com",
  "instance_fingerprint": "5f3c2a7b9d1e4f6a8c0b2d4e6f8a1c3d5e7f9a0b1c2d3e4f5a6b7c8d9e0f1a2b",
  "hostname": "appliance-01",
  "ip": "10.0.0.12"
}
```

响应体：
```json
{
  "code": "0",
  "message": "success",
  "data": {
    "appliance_id": "appl_01JABCXYZ",
    "is_new": true
  }
}
```

返回码建议：
1. `200`：成功。
2. `403`：AK/SK 校验失败。
3. `A409`：指纹冲突（极端场景，需人工处理）。
4. `A429`：请求过频。

。

## 五、安全
1. SK 只接收不回传，日志中统一脱敏。
2. 服务端使用 KMS 或等价方案保存 `account_sk_cipher`。


# 改进建议（一个）
引入“实例指纹强约束去重”机制，并将其作为数据库唯一约束（`uk_account_fingerprint`）。

价值：
1. 彻底解决网络重试导致的重复注册问题。
2. 解决同一账号下多实例的身份歧义问题。
3. 让重装恢复场景可控，降低运维排障成本。

## `instance_fingerprint` 生成流程
1. 采集原始机器标识：
   - Linux：优先 `/etc/machine-id`，缺失时退化到主板序列号或系统 UUID。
   - Windows：优先 `MachineGuid`（注册表），缺失时退化到 BIOS UUID。
2. 首次安装生成 `install_salt`（128 bit 随机值），写入本地配置文件（仅本机可读）。
3. 按固定格式拼接原文：`fingerprint_source = domain + "|" + machine_id + "|" + install_salt`。
4. 计算摘要：`instance_fingerprint = SHA-256(fingerprint_source)`，落库/上传时使用十六进制字符串。
5. 本地持久化：保存 `install_salt` 与 `instance_fingerprint`，服务重启直接复用，不重复生成。
6. 重装策略：
   - 同机重装且保留数据目录：沿用原 `install_salt`，得到相同指纹，可恢复原 `appliance_id`。
   - 同机重装未保留数据目录：会生成新指纹，云侧按新实例注册。
7. 校验规则：
   - 长度固定 64（hex），不合法直接拒绝注册请求。
   - 禁止上传明文 `machine_id`，只上传 `instance_fingerprint`。
